How BIOTP Works

Protocol details

Overview

BIOTP replaces the symmetric shared secret in TOTP with per-time-step ECDH key agreement over P-256. The device private key lives in a hardware secure element (Secure Enclave on iOS, TEE/StrongBox on Android) and is gated on biometric authentication. The server stores only user public keys. OTP generation is fully offline after a one-time registration.

1. Registration

The device scans a QR payload (or pastes it manually). The payload contains:

  • callback_url: full URL the device POSTs registration to
  • master_public: server master public key, raw P-256 X||Y (64 bytes)
  • period: time step in seconds (default 30)
  • session_id: unique registration session
  • rp_name: relying party display name
  • attest_challenge: random challenge for device attestation

The device generates a P-256 keypair in the secure element and POSTs its public key, platform identifier, and attestation material to the callback_url.

2. Child Key Derivation

For counter c = floor(unix_time / period), both sides derive matching child keys using the master public key as the HMAC key: 

tweak        = HMAC-SHA256(master_public, uint64_be(c)) mod n
child_public  = master_public + tweak * G        // device computes this
child_private = (master_private + tweak) mod n    // server computes this

n is the P-256 curve order. G is the generator point. The tweak is publicly computable — its purpose is freshness, not secrecy.

3. OTP Generation

After biometric authentication unlocks the secure element key, the device computes: 

shared_secret = X9.63-KDF-SHA256(ECDH(device_private, child_public), L=32)
h             = HMAC-SHA256(shared_secret, uint64_be(c))
offset        = h[31] & 0x0F
otp           = trunc31(h[offset..offset+4]) mod 1_000_000

This is standard RFC 4226 dynamic truncation with SHA-256 instead of SHA-1.

4. Server Verification

The server checks the submitted OTP against the current, previous, and next counter:

  1. Derive child_private for each candidate counter.
  2. Compute ECDH with the stored user public key.
  3. Apply X9.63-KDF-SHA256 (32-byte output).
  4. Recompute the 6-digit OTP and compare in constant time.
  5. Reject if the counter was already used (replay protection).
5. Device Attestation

During registration, the device provides a hardware attestation proving the key is held in a genuine secure element. The format is platform-specific:

iOS — Apple App Attest

  • CBOR attestation object from DCAppAttestService
  • x5c certificate chain → Apple App Attestation Root CA
  • Nonce binding via OID 1.2.840.113635.100.8.2
  • Optional rpIdHash enforcement (Team ID + Bundle ID)

Android — KeyStore Attestation

  • X.509 DER certificate chain from KeyStore
  • Chain verified → Google hardware attestation root
  • Key attestation extension OID 1.3.6.1.4.1.11129.2.1.17
  • Attestation challenge embedded at key generation
Security Properties
  • No per-user secrets on server. Only public keys are stored. A database breach exposes nothing useful.
  • Device-bound. The private key is non-extractable from the secure element. Cloning the authenticator is not possible.
  • Biometric-gated. Every OTP generation requires a live biometric match enforced by hardware. Malware cannot silently generate codes.
  • Offline. No network needed after registration.
  • Standard output. 6-digit code compatible with any OTP input field.
  • Trade-off: The server master private key is a single high-value secret. If compromised together with the user database, an attacker can generate OTPs. HSM protection is recommended.
Note

This server is a reference implementation for demonstration purposes. It uses in-memory storage — restarting clears all users and sessions.
